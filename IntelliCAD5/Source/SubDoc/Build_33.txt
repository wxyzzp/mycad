Reference:      BugZilla:78700

Description:    This submission fixes the problem - BREAK command keeps asking 
                for selection of second entity after the second break point is 
                specified, when the intersect mode of Osnap is ON. 

                The current implementation of BREAK uses sds_getpoint() to 
                prompt users to specify the second break point. When the users 
                pick a point on the selected entity and the Intersect Osnap is 
                active, sds_getpoint() does not return a point but instead it 
                prompts users to select second entity until either users select 
                the second entity or enter ESC key. The root of the problem is 
                actually in the Intersect Osnap code, which incorrectly 
                implements Intersect Osnap in such way that once an entity is 
                selected to get point on the entity it will ask users to select 
                second entity. This behavior is wrong because it doesnít work 
                in the case that users just want to get a point on an entity 
                that does not intersect with any entity. This problem is fixed.

Description:    This submission fixes the problem - exploding a circle block 
                inserted with non-uniform scale is rotated results in a huge 
                rotated ellipse.

                by those independent scale values.

Description:

This submission completely fixes BHATCH and BOUNDARY reported in the bug #78655 
and # 78640. The most BHATCH and BOUNDARY related problems have actually been 
fixed automatically when other problems were fixed in the early submission 
except the following two problems, which are fixed by this submission.

1) BHATCH does not work when the boundary of selected area is made up by 
   spline(s) with other spline or non-spline curves. The problem causing BHATCH 
   to fails is the algorithm implemented in CNURBS:: ReplaceWithRestriction?() 
   function is not good and reliable so it fails often when it is called to get 
   a sub-curve of b-spline to construct the boundary of  hatch.  This bug is 
   fixed by replacing existing implementation of CNURBS:: ReplaceWithRestriction?() 
   with a call to Subset() function to get the sub-curve.

2) BOUNDARY does not work for lines. The root of problem is in 
   IsCurvePolylineComponent?() function because it lacks support to 
   TYPE_UnboundedLine and TYPE_Ray  so the function incorrectly identify a 
   CUnboundedLine? or CRay? curves can not be converted to a component of polyline 
  (line or arc). This problem is fixed by modifying IsCurvePolylineComponent?() 
   to add the TYPE_UnboundedLine and TYPE_Ray cases.

Description:	

This submission fixes the following TRIM problems:

1)Unable to trim lines, xlines, rays, circles/arcs, light weight polylines, 
  and polylines, using ellipses or splines because the functionality was not 
  implemented. This submission adds the implementation for those entities 
  with ellipses and splines.

2)Trimming closed and/or periodic splines and ellipses did not give correct 
  results when the remained part across the seam. The result was either two 
  separated pieces or was not shown correctly. Those problems have fixed by 
  adding code to join the remained pieces together and get a sub-curve of 
  spline.

3)The splines and ellipses were not trimmed correctly when the multiple curves 
  were used as cutting entities. The original implementation did not consider 
  multiple that one cutting curs so only the first curve in the selection set 
  was used to trim splines and ellipses. This problem is fixed.

4)The closed ellipses were not trimmed correctly when any of cutting curves 
  pass the seam.  In that case, the start or end point  of the ellipse is a 
  trimming point too but those special cases were not taken account and handled 
  specially. Those cases are taken care in this submission.

5)Trimming elliptic arcs did not work correctly when the end parameter of the 
  elliptic arc was smaller than the start parameter. In that case, 
  CEllipArc?::SetDomain?() would reject to set domain of the curve, which caused 
  a full ellipse to be used to compute the intersection with other curves so 
  the trimming points were not located in the boundary of the elliptic arc. 
  This problem is fixed by adding 2PI to the end parameter if it is smaller 
  than start parameter.

6)Using Xlines and rays to trim other curves or using other curves to trim Xlines 
  and rays did not give correct results somtimes. The problems was xlines and rays 
  were converted to CLine? rather than CUnboundedLine? and CRay? in the code so the 
  intersections were not found somtimes. This problem is corrected.

7)Trimming a crossing polyline with itself resulted in a polyline without selected 
  segment, which is wrong totally. The problem was the underlying function to take 
  care this special case was not implemented correctly. It just simply deleted the 
  selected segment from polyline instead of trim polyline at the self-intersecting 
  points. This problem is fixed by eliminating that function to treat the case like 
  regular polyline trimming polyline case.

8)Unable using Undo command to rollback last the trimming operation after ellipses 
  and splines were trimmed. The problem was the sds_entupd() rather than using 
  sds_entmod() was used to update the changes of ellipses and splines so the changes 
  were never written to the undo stack. This problem is fixed by modify the code to 
  update modified ellipse and splines by using sds_entmod() function.

9)The curves were not trimmed correctly when edge mode is on. The reasons was the 
  unbounded geometries  of the cutting and trimmed entities was not used to compute 
  the intersections  in some cases when edge mode is on so the results was not correct. 
  This submission also fix this problem.

This submission also fix the problem in CNURBS::JoinWith?() function in GEO library, 
which did not join splines correctly. But this submission doesnít intend to any 
address issue related to projected trimming.  Trimming the line an arc in UCS is 
not implemented correctly and is not implemented for other curves and trimming any 
curve in viewing direction is not implemented. The EXTEND command also has similar 
problems. To perform projected trimming, we need to have funcitonality to project 
all curves along an arbitrary direction to an arbitrary plane and compute the 
intersection of C0 continuity spline curves(the curves could be C0-continuity after 
projected). Currently there are not such functions in IntelliCAD? code. All those 
issues will be addressed in my later submission.
